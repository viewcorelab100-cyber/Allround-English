# Plan: Demo Account (읽기 전용 데모 계정)

## 1. 배경 및 목적

### 문제 정의
- 다른 학원 원장님들이 Allround English 시스템의 레퍼런스를 보고 싶어함
- 현재 관리자 계정은 접근 제한이 걸려 있어 공유 불가
- 관리자 계정을 직접 주면 데이터 수정/삭제 등 보안 위험 존재

### 목표
- **읽기 전용 데모 계정**을 만들어 외부 원장님이 시스템을 직접 체험할 수 있게 함
- 실제 학생 데이터를 노출하지 않고, 더미 데이터로 구성된 환경을 제공
- 기존 관리자 기능과 완전히 분리된 안전한 열람 경험 제공

## 2. 현재 시스템 분석

### 인증 구조
- **Supabase Auth** 기반 이메일/비밀번호 인증
- `profiles` 테이블의 `role` 컬럼으로 권한 구분 (`admin` / 일반)
- `isAdmin()` 함수: `profile.data.role === 'admin'` 체크
- 관리자 페이지(`admin.html`): `checkAdminAccess()` → `isAdmin()` → 실패 시 `index.html`로 리다이렉트

### 세션 관리
- 최대 3대 기기 동시 접속 제한
- `active_sessions` 배열로 세션 추적
- 주기적 세션 검증 (30초 간격)

### 관리자 페이지 기능
- 학생 관리 (프로필 조회, 수정)
- 강의 관리 (CRUD)
- 레슨 관리
- 결제/매출 관리
- 진도 데이터 조회

## 3. 솔루션 설계

### 방안: `demo` 역할 추가 + 읽기 전용 관리자 페이지

#### 3.1 역할 체계 확장

| 역할 | 값 | 권한 |
|------|-----|------|
| 일반 학생 | `student` (기본) | 본인 강의만 접근 |
| 관리자 | `admin` | 전체 CRUD |
| **데모** | **`demo`** | **관리자 페이지 읽기 전용** |

#### 3.2 핵심 구현 사항

**A. 데모 계정 생성**
- Supabase에 `demo@allround.com` 계정 생성 (또는 관리자가 원하는 이메일)
- `profiles.role = 'demo'` 설정
- 비밀번호는 관리자가 설정/공유

**B. 권한 분리 (프론트엔드)**
- `isAdmin()` → `isAdminOrDemo()` 확장 (관리자 페이지 접근 허용)
- `isDemoUser()` 함수 추가 (쓰기 작업 차단용)
- 관리자 페이지에서 데모 계정일 경우:
  - 모든 **생성/수정/삭제 버튼 숨김** 또는 비활성화
  - **데이터 조회(읽기)만 허용**
  - 상단에 "데모 모드" 배너 표시

**C. 권한 분리 (Supabase RLS)**
- `demo` 역할은 `SELECT`만 허용하는 RLS 정책 추가
- `INSERT`, `UPDATE`, `DELETE`는 `role = 'admin'`만 허용

**D. 더미 데이터**
- 실제 학생 데이터 대신 더미 데이터 표시 여부는 **선택적**
- 방안 1: 실제 데이터를 읽기 전용으로 보여줌 (간단, 실제감 있음)
- 방안 2: 별도 더미 데이터셋 구성 (보안 강화, 추가 개발 필요)
- **추천: 방안 1** (실제 데이터 읽기 전용 → 개발 비용 최소화, 학생 개인정보는 마스킹 처리)

#### 3.3 데이터 마스킹 (개인정보 보호)

데모 계정으로 볼 때 민감 정보 마스킹:

| 필드 | 원본 | 마스킹 |
|------|------|--------|
| 이름 | 김민수 | 김** |
| 이메일 | student@gmail.com | stu***@gmail.com |
| 전화번호 | 010-1234-5678 | 010-****-5678 |
| 보호자 전화 | 010-9876-5432 | 010-****-5432 |
| 주소 | 서울시 강남구 ... | 서울시 강남구 *** |

## 5. 기술적 리스크

| 리스크 | 심각도 | 대응 |
|--------|--------|------|
| 프론트엔드만 숨기면 API로 직접 쓰기 가능 | 높음 | **반드시 RLS로 서버 측에서도 차단** |
| 실제 학생 개인정보 노출 | 중간 | 마스킹 처리로 대응 |
| 데모 비밀번호 유출 | 낮음 | 주기적 비밀번호 변경, 필요 시 계정 비활성화 |
| 데모 계정의 세션 제한 (3대) | 낮음 | 데모 계정은 세션 제한 완화 또는 별도 처리 |
